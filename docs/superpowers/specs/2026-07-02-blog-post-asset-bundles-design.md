# 글별 이미지 번들 설계

## 목표

블로그 글과 그 글에서 사용하는 커버·본문 이미지·GIF를 하나의 날짜 기반 폴더에서 함께 관리한다. 폴더명은 사람이 파일 탐색기에서 날짜순으로 정리하기 위한 용도이며, 공개 URL은 frontmatter의 고정 `slug`를 사용한다.

## 디렉터리 구조

모든 글은 다음 구조를 따른다.

```text
src/content/blog/<category>/<YYYY-MM-DD 자유로운 제목>/
├── index.md
└── images/
    ├── cover.webp
    ├── architecture.png
    └── demo.gif
```

폴더명의 날짜 뒤 제목은 한글, 영문, 공백을 허용한다. 예시는 다음과 같다.

```text
src/content/blog/backend/2026-07-02 FastAPI 비동기 작업 처리/
```

첫 번째 경로 조각인 `backend`는 카테고리다. 두 번째 경로 조각은 날짜 기반 관리 폴더다. `index.md`만 콘텐츠 엔트리로 읽고 `images/` 안의 파일은 Astro 자산으로 처리한다.

폴더명은 `YYYY-MM-DD `로 시작해야 한다. 이 규칙을 어기면 테스트와 빌드 검증에서 오류로 알려준다. 화면에 표시하고 정렬할 날짜는 frontmatter의 `date`를 사용하며, 폴더 날짜와 `date`는 같아야 한다.

## Frontmatter와 URL

각 글은 고정 URL을 위한 `slug`를 필수로 작성한다.

```yaml
---
title: FastAPI 비동기 작업 처리
date: 2026-07-02
slug: fastapi-background-task
desc: FastAPI의 비동기 작업 구성 방법을 정리합니다.
lead: API 응답과 장기 실행 작업을 분리하는 기준을 살펴봅니다.
tags: [Python, FastAPI, Backend]
readTime: 7 MIN
fresh: true
cover: ./images/cover.webp
coverAlt: FastAPI 비동기 작업 처리 구조
---
```

공개 주소는 폴더명이 아니라 `slug`로 만든다.

```text
/blog/backend/fastapi-background-task/
```

`slug`는 URL 안정성을 위해 소문자 영문·숫자·하이픈만 허용한다. 글의 폴더명이나 게시일이 바뀌어도 `slug`가 같으면 URL은 유지된다. 같은 카테고리 안에서 중복된 slug는 허용하지 않는다.

## 이미지와 GIF

본문에서는 Markdown 상대경로를 사용한다.

```md
![전체 처리 구조](./images/architecture.png)

![실행 화면](./images/demo.gif)
```

`src/` 아래의 Markdown 상대 이미지는 Astro가 빌드 자산으로 포함한다. PNG, JPEG, WebP 등 정적 이미지는 Astro 이미지 파이프라인의 최적화를 사용한다. GIF는 지원되는 로컬 입력 형식으로 취급하고 애니메이션을 유지하며, 코드에서 정적 WebP로 강제 변환하지 않는다.

모든 Markdown 이미지에는 의미 있는 대체 텍스트를 작성한다. 장식용 이미지는 빈 대체 텍스트를 사용할 수 있다.

## 커버 이미지

`cover`와 `coverAlt`는 함께 사용하는 선택 필드다.

- 두 필드가 모두 없으면 현재 텍스트 중심 레이아웃을 유지한다.
- `cover`가 있으면 `coverAlt`도 반드시 있어야 한다.
- `coverAlt`만 단독으로 작성할 수 없다.
- 커버 경로는 글 폴더 기준 상대경로이며 `images/` 안의 파일을 가리킨다.

콘텐츠 컬렉션 스키마는 Astro의 `image()` helper로 커버 파일을 검증하고 이미지 메타데이터로 변환한다.

글 목록에서는 커버를 작은 반응형 썸네일로 표시한다. 상세 페이지에서는 제목과 소개문 아래, 본문 위에 넓은 커버를 표시한다. GIF 커버도 허용하지만 용량이 큰 애니메이션은 본문 이미지로 사용하는 것을 권장한다.

## 기존 글 마이그레이션

기존 네 글은 내용을 변경하지 않고 다음처럼 이동한다.

```text
backend/notes-on-astro.md
→ backend/2026-05-02 Astro를 선택한 이유/index.md
```

각 기존 글에 현재 URL과 같은 `slug`를 추가한다. 기존 글에는 임의 커버 이미지를 만들지 않는다. 따라서 마이그레이션 후에도 현재 글 목록과 상세 화면의 텍스트 레이아웃이 유지된다.

## 코드 변경

- 콘텐츠 스키마에 필수 `slug`, 선택적 `cover`, 선택적 `coverAlt`와 두 커버 필드의 쌍 검증을 추가한다.
- 글 ID 파서는 `category/<dated-folder>/index.md` 구조에서 카테고리를 읽고 `entry.data.slug`를 공개 slug로 사용한다.
- 폴더 날짜 형식과 frontmatter 날짜 일치를 검사하는 순수 함수를 추가한다.
- 목록과 상세 컴포넌트에 선택적 커버 렌더링을 추가한다.
- 본문과 커버 이미지가 작은 화면 너비를 넘지 않도록 공통 반응형 스타일을 적용한다.

## 오류 처리

- 날짜 접두사가 없는 글 폴더는 검증 실패로 처리한다.
- 폴더 날짜와 frontmatter 날짜가 다르면 검증 실패로 처리한다.
- 잘못된 slug 형식이나 같은 카테고리의 중복 slug는 검증 실패로 처리한다.
- 커버 파일이 없거나 `cover`와 `coverAlt` 중 하나만 있으면 콘텐츠 스키마 검증이 실패한다.
- 본문에서 존재하지 않는 상대 이미지를 참조하면 Astro 빌드 실패를 그대로 노출한다.

## 검증

자동 테스트는 다음을 확인한다.

1. 날짜·한글·공백이 포함된 폴더에서도 카테고리와 frontmatter slug를 올바르게 추출한다.
2. 날짜 폴더 형식과 frontmatter 날짜 일치를 검증한다.
3. 카테고리별 slug 중복을 탐지한다.
4. 커버 스키마와 선택적 렌더링 계약이 존재한다.
5. 기존 네 글이 날짜 폴더의 `index.md`로 이동하고 기존 공개 URL을 유지한다.
6. 전체 정적 빌드가 카테고리 목록과 상세 페이지를 생성한다.

브라우저에서는 커버 없는 기존 글의 레이아웃과 모바일 가로 넘침을 확인한다. 기존 글에 임의 이미지를 추가하지 않으므로, 선택적 커버 렌더링은 스키마·컴포넌트 소스 테스트로 검증하고 사용자가 실제 커버를 추가했을 때 전체 빌드가 파일 경로까지 검증하도록 한다.

## 범위 제외

이미지 편집·리사이징 UI, 업로드 화면, 외부 이미지 CDN, 영상 파일, 날짜별 아카이브 페이지는 이번 변경에 포함하지 않는다.
